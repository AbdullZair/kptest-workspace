# ADR-007: EAS Build pipeline — mobile distribution (iOS + Android)

## Status

**ACCEPTED** — 2026-04-29

## Kontekst

User Story **US-S-14** wymaga, aby aplikacja KPTEST Mobile mogła być budowana
i dystrybuowana na obu platformach mobilnych (iOS + Android) przez kanały
testowe (TestFlight, Google Play Internal Testing) i docelowo produkcyjne.

Aktualny stan repozytorium `mobile/`:

- React Native 0.73 + Expo SDK 50 (managed workflow).
- Brak skonfigurowanego pipeline build/distribute — `mobile-ci.yml` zawiera
  jedynie krok `eas build:configure` jako placeholder; brak `eas.json`,
  brak workflowu `eas-build.yml`.
- Brak generated native code (`ios/`, `android/`) — projekt pozostaje w
  workflow Expo managed, więc Fastlane wymagałby najpierw `expo prebuild`
  i utrzymywania natywnych projektów Xcode/Gradle.
- Zespół mobile używa już Expo SDK i konta Expo do dev clienta — naturalna
  ścieżka do EAS Build (ten sam ekosystem).

**Rozważane opcje**:

1. **Fastlane** (Ruby) — klasyczne narzędzie iOS/Android. Wymaga `expo prebuild`,
   utrzymywania natywnych projektów, własnej konfiguracji code signing
   (match/sigh), Fastfile dla każdej platformy. Wysoka kontrola, wysoki koszt
   utrzymania.
2. **EAS Build (Expo Application Services)** — zarządzany build cloud Expo.
   Wbudowane code signing, automatyczne `app.json` → native build, integracja
   z `expo-updates` (OTA), `eas submit` do TestFlight/Play. Free tier
   (~30 buildów/miesiąc) wystarczający dla jednego zespołu.
3. **GitHub Actions + macOS runner + xcodebuild/gradle** — własnoręczna
   automatyzacja. Wymaga macOS runnera (płatny: $0.08/min), własnego
   zarządzania certyfikatami w GitHub Secrets. Maksymalna kontrola, najwyższy
   koszt setupu.
4. **Bitrise / CircleCI mobile** — komercyjne CI mobile-first. Bardziej rozwinięte
   niż GHA, ale oddzielny vendor i koszt.

## Decyzja

**Wybieramy opcję 2 — EAS Build (Expo Application Services)**:

- Trzy profile build w `mobile/eas.json`:
  - `development` — dev client (iOS simulator + Android debug APK).
  - `preview` — internal testers (Android APK + iOS device build),
    `API_BASE_URL=https://staging-api.kptest.com`.
  - `production` — Play Store (`app-bundle`) + App Store (`autoIncrement`),
    `API_BASE_URL=https://api.kptest.com`.
- `eas submit` → Google Play Internal Testing track + App Store Connect
  TestFlight.
- GitHub Actions workflow `.github/workflows/eas-build.yml`:
  - Trigger automatyczny: push na `main` (zmiany w `mobile/**`) → preview build.
  - Trigger manualny: `workflow_dispatch` z wyborem profile (dev/preview/prod)
    i platform (all/ios/android).
  - Wymagany sekret `EXPO_TOKEN` w GitHub Secrets.

Motywacja:

- **Najmniejszy setup** — Expo managed workflow + EAS Build = brak
  utrzymania natywnych projektów Xcode/Gradle.
- **Code signing managed** — EAS przechowuje certyfikaty/profiles iOS, keystore
  Android; rotacja przez `eas credentials`.
- **Integracja z `expo-updates`** — OTA updates (channels `preview`/`production`)
  bez nowego buildu store'owego.
- **Niski koszt** — free tier EAS pokrywa development pace zespołu (≤30
  buildów/mies.); przy większym wolumenie plan Production ($99/mies.).
- **CI/CD-first** — `eas build --non-interactive --no-wait` zwraca natychmiast,
  build zachodzi w cloudzie Expo (oszczędność minut macOS runner).

## Konsekwencje

### Pozytywne

- **Time to first build** ≈ kilka minut (vs. dni dla Fastlane setup).
- **Brak macOS runner** w GHA → niższy rachunek + brak utrzymania.
- **Single source of truth** dla wersjonowania (`appVersionSource: remote`
  w `eas.json` — EAS zarządza `buildNumber`/`versionCode`).
- **TestFlight + Play Internal Testing** dostępne natychmiast po `eas submit`.
- **Channels** w `expo-updates` (preview/production) → OTA bez store review
  dla zmian JS-only.

### Negatywne

- **Vendor lock-in (Expo)** — EAS Build jest zamknięty (managed cloud).
  *Mitigation*: `expo prebuild` w każdej chwili wygeneruje natywne `ios/` +
  `android/` projects (eject pathway pozostaje otwarty). Konfiguracja
  `app.json` jest standardem Expo i nie blokuje migracji.
- **Free tier limits** — 30 buildów/miesiąc, kolejka shared. *Mitigation*:
  Production plan ($99/mies.) gdy zespół rośnie, lub self-hosted EAS Build
  (jeśli compliance tego wymaga).
- **EXPO_TOKEN secret** — kompromitacja tokenu = atak na build pipeline.
  *Mitigation*: token scoped per CI, rotacja co 90 dni, GitHub Secrets
  audit log, `Restrict to organization` na poziomie Expo team.
- **Apple Developer + Google Play konta** — koszt $99/rok + $25 jednorazowo,
  zarządzane przez człowieka (KYC, payment, agreements). EAS nie automatyzuje
  tych zakupów.
- **Generated config drift** — `extra.eas.projectId` musi być wstrzyknięty
  przez `eas build:configure` (TODO: po pierwszym uruchomieniu nadpisze
  placeholder UUID `00000000-...` w `app.json`).

### Plan łagodzenia

- **Token rotation runbook** — `docs/operations/expo-token-rotation.md` (TODO follow-up).
- **Build budget alerting** — Expo dashboard email gdy ≥80% miesięcznego limitu.
- **Eject pathway** — utrzymujemy `mobile/app.json` jako single source of truth
  (zamiast modyfikować generated `ios/`/`android/`), żeby `expo prebuild`
  pozostał deterministyczny.
- **Per-environment env vars** — `eas.json` `env.API_BASE_URL` per profile
  (`preview` vs `production`); secrets (np. Sentry DSN) przez `eas secret:create`.

## Re-ewaluacja

ADR powinien zostać przejrzany jeśli:

- Zespół potrzebuje bardziej skomplikowanych etapów native (np. własny C++
  toolchain, custom CocoaPods source) niż EAS Build wspiera.
- Free tier przestaje wystarczać i koszt Production plan nie jest akceptowalny
  w porównaniu z self-hosted runner.
- Pojawi się compliance wymóg (HIPAA/RODO) zabraniający przechowywania
  source code i certyfikatów w cloud Expo — wtedy migracja na self-hosted
  EAS Build / GitHub Actions + macOS runner.
- Expo zmieni model licencyjny lub porzuci managed workflow.

## Implementacja (C2)

### Pliki

- `mobile/eas.json` — 3 profile (development / preview / production) + submit
  config (Play `internal` track + TestFlight).
- `mobile/app.json` — `runtimeVersion: { policy: 'appVersion' }`,
  `extra.eas.projectId` placeholder (nadpisany przy `eas build:configure`),
  `updates.url` placeholder.
- `.github/workflows/eas-build.yml` — push/PR/dispatch triggers,
  `expo/expo-github-action@v8`, `EXPO_TOKEN` secret.
- `mobile/EAS_BUILD.md` — runbook dla developerów.

### Verification (manual, wymaga konta Expo)

- `cd mobile && eas build:configure` → `app.json.extra.eas.projectId` UUID.
- `eas build --profile preview --platform android` → APK URL.
- `eas submit --platform android` → Google Play Internal Testing track.
