# ADR-006: Mobile E2E — Maestro zamiast Detox

## Status

**ACCEPTED** — 2026-04-29

## Kontekst

Projekt KPTEST mobile (React Native 0.73 + Expo SDK 50, target
`com.kptest.mobile`) zawiera **23 historie pacjenta US-P-01..23**
(rejestracja, logowanie, 2FA, projekty, materiały, kalendarz, powiadomienia,
compliance) i do dziś nie posiada żadnych testów End-to-End. Memory wpisu
`project_implementation_gaps.md` oraz raport `MOBILE_COMPLETE_REPORT.md`
oznaczają tę powierzchnię jako "100 % feature complete, 0 % E2E coverage".

W trakcie audytu C1 rozważono dwa pomocnicze frameworki:

### Opcja A — Detox (standard React Native)

- **Zalety**: oficjalna rekomendacja dla CLI'owego React Native, white-box,
  szybkie synchronizacje (`waitFor`/`expect`), dojrzały (5 lat).
- **Wady przy Expo Managed Workflow**:
  - Wymaga `expo prebuild` (eject do bare workflow), co dodaje katalogi
    `android/` i `ios/` do repo i odbiera korzyści Managed (OTA, EAS Build).
  - Konfiguracja `detoxrc.js` + `applesimutils` + `xcodebuild`/`gradlew` dla
    obu platform (~6h setupu).
  - Test runner = Jest, scenariusze pisane w TS — duplikacja konwencji
    z testów jednostkowych.
- **Czas setupu (estymata)**: 1.5–2 dni dev.

### Opcja B — Maestro (mobile.dev)

- **Zalety**:
  - Zero modyfikacji projektu Expo — działa na zbudowanym IPA/APK
    (`npx expo run:android` wystarczy).
  - Scenariusze deklaratywne **YAML** (czytelne dla QA/PM, brak typescriptu).
  - Wbudowany inspektor (`maestro studio`) podgląda hierarchię UI.
  - Cross-platform syntactically: ten sam YAML chodzi na Androidzie i iOS.
  - JUnit reporter wbudowany, stabilne CI z `reactivecircus/android-emulator-runner`.
- **Wady**:
  - Niższa precyzja niż Detox (asercje text/accessibility, brak white-box).
  - Brak natywnego mockowania API (trzeba przez backend dev/test environment).
  - Maestro CLI dystrybuowane przez bash installer (nie przez npm).
- **Czas setupu**: 2–4h (zrobione w ramach C1).

## Decyzja

**Wybieramy opcję B — Maestro.**

Maestro pozostawia projekt w Expo Managed Workflow (ważne dla EAS Build /
OTA), redukuje krzywą wejścia dla QA (YAML zamiast TypeScript) i ma
niższy łączny koszt utrzymania dla zespołu rozmiaru KPTEST. Brak
white-boxa rekompensuje fakt, że testujemy black-box akcept-criteria
z perspektywy pacjenta (US-P-01..23 to user journeys, nie internal state).

Zakres iteracji 1 (C1):

- **5 zaimplementowanych flow YAML**: US-P-01 (rejestracja), US-P-03
  (logowanie), US-P-10 (lista projektów), US-P-15 (materiały),
  US-P-18 (kalendarz). Pliki w `mobile/maestro/flows/p{nn}-{slug}.yaml`.
- **18 pozostałych USes** zostawione jako TODO z opisanymi krokami
  i dependencies w `mobile/maestro/flows/_todo.md`.
- `mobile/package.json` skrypty: `test:e2e`, `test:e2e:single`,
  `test:e2e:report` (JUnit).
- `mobile/maestro/README.md` z instrukcją lokalnego uruchomienia
  (Android emulator + backend docker compose).

## Konsekwencje

### Pozytywne

- **Brak ejecta** — projekt pozostaje 100 % Managed Expo, co utrzymuje
  ścieżkę EAS Build, EAS Update OTA i Hermes auto-config.
- **Niski próg wejścia QA** — scenariusze YAML czytelne dla osób bez
  background-u w React Native.
- **Cross-platform syntactically** — jeden YAML, dwie platformy.
- **5 krytycznych flow happy-path** zabezpieczonych już w C1 (smoke).
- **Reporter JUnit** — łatwa integracja z istniejącym pipeline'em
  Playwright JUnit (`tests/test-results.xml`).

### Negatywne

- **Brak białoskrzynkowej introspekcji** Redux state/ AsyncStorage —
  asercje wyłącznie po UI. Krytyczne edge-case'y (np. token refresh)
  trudniejsze do zweryfikowania (mitigation: jednostki + integracja
  RTK Query w Jest).
- **Maestro CLI poza npm** — instalacja przez bash installer
  (mitigation: README z jednoliniowcem; CI może instalować w workflow).
- **18/23 scenariuszy TODO** — coverage 22 % po C1, plan dokończenia
  w iteracjach C2..C5 (target: 100 % do końca FAZY 4).
- **Zależność od `testID`** — stabilność scenariuszy wymaga, by ekrany
  konsekwentnie eksponowały `testID` (issue M-TEST-001 zaplanowany).
- **Sandbox w obecnej iteracji** — Maestro nie został uruchomiony
  (brak emulatora Android w sandboxie agenta); scenariusze zweryfikowane
  wyłącznie syntaktycznie. Pierwszy lokalny run = developer task.

### Plan łagodzenia

- **CI follow-up**: `.github/workflows/mobile-e2e.yml` z
  `reactivecircus/android-emulator-runner@v2` (API 33), uruchamia
  `npx expo run:android` + `maestro test maestro/flows/` z JUnit reportem
  uploadowanym do `actions/upload-artifact`.
- **Seed E2E** (`backend/sql/seed-e2e.sql`): pacjent demo
  (`patient.demo@kptest.example.com` / `DemoP@ssw0rd123`), projekt
  aktywny, materiał, wydarzenie kalendarza.
- **`testID` audit** (issue M-TEST-001): dodanie identyfikatorów do
  LoginScreen, RegisterScreen, ProjectsScreen, MaterialsScreen,
  CalendarScreen zgodnie z `mobile/CLAUDE.md`.
- **Iteracje C2..C5**: po 4–5 dodatkowych flow per iteracja, dotarcie
  do 23/23 do końca FAZY 4.

## Re-ewaluacja

ADR powinien zostać przejrzany jeśli:

- Projekt zrezygnuje z Expo Managed (eject do bare RN) — wówczas Detox
  staje się ponownie atrakcyjny (white-box).
- Pojawi się wymóg sterowania natywnym API (np. push tokens, biometric
  prompt) na poziomie scenariusza E2E, którego Maestro nie obsłuży —
  wtedy hybryda Detox + Maestro lub przejście na Detox.
- Zespół QA przekroczy 5 osób i pojawi się potrzeba pełnego TS API
  (Page Object Model) — Detox lepiej skaluje się dla dużych zespołów.
