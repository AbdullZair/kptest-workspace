# ADR-004: OAuth2 Login skeleton in SecurityConfig

## Status

**ACCEPTED** — 2026-04-29

## Kontekst

Audyt P5d (extension-points.md §6, §7.1) wykrył, ze:

- `com.kptest.infrastructure.config.SecurityConfig` nie zawiera `OAuth2LoginConfigurer`.
- `com.kptest.application.service.AuthenticationService` ma jeden hardcoded flow:
  `PasswordEncoder` + opcjonalny `TotpService`.
- Brak interfejsu `AuthenticationProvider` na poziomie aplikacji.

Konsekwencja: dodanie SSO (Google Workspace, Microsoft Entra ID / Azure AD,
Keycloak) wymagaloby refaktoru `SecurityConfig`, dodawania dependencies do
`build.gradle`, tworzenia `application-oauth2.yml` od zera oraz testow integracyjnych.
Jest to klasyczny **dlug architektoniczny** typu "missing extension point".

User Story `US-S-05` ("OAuth2 / SSO support") byla od dawna planowana jako
follow-up — aktualnie pierwszy krok w tym kierunku.

### Rozwazane opcje

1. **Pelna implementacja OAuth2** — `GoogleOidcUserService`, `OAuthLoginSuccessHandler`,
   tabela `user_external_identity`, mapping `OidcUser` → lokalny `User`. Duzy zakres
   (tydzien+ pracy), wymaga decyzji produktowych (auto-create vs invite-only,
   allowed domains, role mapping).
2. **Tylko dependencies + szkielet konfiguracji** — dodac jary OAuth2,
   `Customizer.withDefaults()` w `SecurityConfig`, profil `application-oauth2.yml`
   z placeholderami. Aktywacja przez env vars w przyszlosci.
3. **Status quo** — odlozyc zupelnie do US-S-05 implementation sprint.

## Decyzja

Wybieramy **opcje 2 — szkielet OAuth2 w SecurityConfig (default disabled)**.

Konkretnie:

1. **Dependencies** w `backend/build.gradle`:
   ```gradle
   implementation 'org.springframework.boot:spring-boot-starter-oauth2-client'
   implementation 'org.springframework.boot:spring-boot-starter-oauth2-resource-server'
   ```

2. **`SecurityConfig.SecurityFilterChain`** rozszerzony o:
   ```java
   .oauth2Login(Customizer.withDefaults())
   ```
   z komentarzem TODO i informacja, ze providery konfiguruje sie przez `application.yml`.

3. **`application-oauth2.yml`** — nowy profil z **wykomentowanymi/placeholder**
   wpisami dla Google (aktywne, ale puste env vars) i Microsoft (zakomentowane).

4. **JWT auth flow pozostaje bez zmian** — `JwtAuthenticationFilter`,
   `AuthenticationService`, `/api/v1/auth/login`, `verify-2fa`, `refresh` dzialaja
   identycznie. OAuth2 to **opcjonalna dodatkowa sciezka logowania**, nie zamiennik.

### Dlaczego `Customizer.withDefaults()`?

Spring Boot autoconfiguruje OAuth2 client **tylko jesli** `spring.security.oauth2.client.registration.*`
zawiera wpisy z niepustym `client-id`. Bez konfiguracji caly graf OAuth2 jest
de facto wylaczony — endpoint `/oauth2/authorization/{registrationId}` zwroci 404,
a `oauth2Login()` configurer nie zarejestruje filterow. Dzieki temu mozemy
"zarezerwowac" miejsce w SecurityFilterChain bez ryzyka regresji.

## Konsekwencje

### Pozytywne

- **Zero-refactor activation**: aktywacja Google OAuth2 to tylko ustawienie
  `GOOGLE_CLIENT_ID` + `GOOGLE_CLIENT_SECRET` env vars i
  `SPRING_PROFILES_ACTIVE=dev,oauth2`.
- **Microsoft / Azure AD / Keycloak**: dodanie kolejnego providera = dopisanie
  bloku `spring.security.oauth2.client.registration.<id>` + `provider.<id>` w yml.
  Bez kodu Java.
- **Documentation alignment**: `extension-points.md` §6 ("BRAKUJACY") moze
  zostac zaktualizowany na "OK — OAuth2 ready".
- **Build security baseline**: oauth2-resource-server wnosi wsparcie dla
  introspekcji JWK i bezpiecznej walidacji tokenow OIDC.

### Negatywne

- **Niedokonczony skrypt mapowania**: `Customizer.withDefaults()` uzywa
  domyslnego `OidcUserService`, ktory **nie tworzy lokalnego `User`** w bazie.
  Po zalogowaniu uzytkownik ma tylko principal w SecurityContext, ale brak
  rekordu `users` ani JWT do API. Pelna implementacja (US-S-05) musi dodac:
  - `GoogleOidcUserService extends OidcUserService` (mapping email → `User`)
  - `OAuthLoginSuccessHandler` generujacy lokalny JWT przez `JwtService`
  - Konfiguracja `oauth2Login(...)` z `userInfoEndpoint(u -> u.oidcUserService(...))`
- **Zwiekszony rozmiar fat-jara**: dwa nowe startery (~3 MB).
- **Default Spring login page**: jesli ktos przypadkowo aktywuje profil bez
  konfiguracji frontendu, `/oauth2/authorization/google` zwroci redirect, ale
  Spring's default login form moze sie pojawic — warto przy aktywacji dodac
  `formLogin(form -> form.disable())` jezeli nie chcemy fallbacku.

### Plan lagodzenia

- Zaplanowac sprintowy ticket "US-S-05 implementation" z zakresem:
  `OidcUserService`, `OAuthLoginSuccessHandler`, tabela `user_external_identity`,
  E2E test `oauth-google.spec.ts`.
- Dopisac sekcje "OAuth2 activation runbook" w `docs/security/` (krok-po-kroku
  jak skonfigurowac Google Cloud Console / Azure App Registration).

## Re-ewaluacja

ADR powinien zostac przejrzany jezeli:

- Wprowadzimy SAML / WS-Federation (inny configurer niz `oauth2Login`).
- Migrujemy do Spring Authorization Server (ksztalt SecurityConfig sie zmieni).
- Dropniemy JWT na rzecz OAuth2 Resource Server jako defaultu (wtedy ADR-001
  rowniez podlega rewizji).
