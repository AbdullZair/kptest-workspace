# ADR-001: Synchronous report export vs async

## Status

**ACCEPTED** — 2026-04-29

## Kontekst

`com.kptest.api.controller.ReportController#exportReport` (`POST /api/v1/reports/export`)
jest zaimplementowany **synchronicznie**: blokuje wątek HTTP, generuje XLSX/PDF w pamięci
przez `ReportService.exportToExcel(...)` / `exportToPDF(...)`, zwraca `ResponseEntity<ByteArrayResource>`
bezpośrednio do klienta.

```java
@PostMapping("/export")
public ResponseEntity<ByteArrayResource> exportReport(@Valid @RequestBody ReportExportRequest req) {
    ByteArrayResource resource = switch (req.format()) {
        case XLSX -> reportService.exportToExcel(reportData, req.reportType().name());
        case PDF  -> reportService.exportToPDF(reportData, req.reportType().name());
    };
    return ResponseEntity.ok().body(resource);
}
```

**Charakterystyka obecnych raportów (FAZA 1–2)**:
- Zakres czasowy: max 12 miesięcy.
- Wiersze: zwykle <5 000 dla raportu compliance, <2 000 dla aktywności pacjenta.
- Rozmiar wynikowego XLSX: zwykle <2 MB.
- Czas generacji: <3 s.

**Rozważane opcje**:

1. **Sync inline** (status quo): generujemy w wątku request-handler, oddajemy `byte[]`.
   - (+) Najprostsze; kompatybilne z curl, Excel-em pobierającym przez AJAX.
   - (-) Wątek HTTP zajęty na cały czas generacji; pamięć JVM rośnie liniowo z rozmiarem.
   - (-) Klient musi czekać aż do końca generacji (timeout reverse-proxy, mobilka traci request przy roamingu).

2. **Async + polling**: `POST /reports/export` zwraca `202 Accepted` z `jobId`,
   klient odpytuje `GET /reports/jobs/{jobId}` aż status zwróci `READY` z URL-em.
   - (+) Skalowalne; backend offloaduje generację na pulę workerów.
   - (-) Wymaga schedulera (Spring `@Async` lub Quartz), tabeli `report_job`,
     mechanizmu czyszczenia.

3. **Async + presigned S3 URL**: jak (2), ale zamiast trzymać plik w bazie/serwerze,
   zapisujemy w S3 i zwracamy presigned URL ważny np. 15 min.
   - (+) Skalowalne, brak pamięci JVM, klient pobiera bezpośrednio z S3.
   - (-) Zależność od S3 (lub MinIO on-prem); presigned URL = wyciekowe ryzyko bezpieczeństwa
     jeśli logujemy URL w access-logach.

4. **Streaming** (`StreamingResponseBody`): generacja na żądanie, chunked transfer.
   - (+) Zero pamięci JVM dla dużych XLSX, jeśli używamy SXSSFWorkbook (Apache POI).
   - (-) Nadal blokuje wątek HTTP; nie pomaga przy timeoutach reverse-proxy >60s.

## Decyzja

**Dla MVP zostajemy przy opcji 1 (sync inline). Migrację do opcji 3 (async + S3 + presigned URL)
uruchamiamy gdy spełniony jest co najmniej jeden trigger:**

- generowany plik przekracza **5 MB** w 95-tym percentylu, **lub**
- czas generacji przekracza **30 s** w 95-tym percentylu, **lub**
- klient zgłasza problemy z timeoutem przy raportach rocznych.

Triggery monitorujemy przez metrykę Prometheus: `kptest_report_export_duration_seconds`
(histogram) i `kptest_report_export_size_bytes` (histogram). Alert Grafany na p95
przekraczające próg.

Implementacja async (gdy uruchomiona):
- Dodać `ReportExportJob` entity (id, userId, status, format, s3Key, createdAt, expiresAt).
- `POST /reports/export` zwraca `202` + `Location: /reports/jobs/{jobId}`.
- Worker z `@Async` (lub Spring Batch) zapisuje wynik do `s3://kptest-reports/{jobId}.xlsx`.
- `GET /reports/jobs/{jobId}` zwraca presigned URL z `expiresIn=900s`.

## Konsekwencje

### Pozytywne
- Brak overengineeringu w MVP.
- Brak zależności od S3/MinIO w pierwszej iteracji (zmniejsza koszt operacyjny i CI/CD).
- Prostota debuggowania — jedna ścieżka kodu, brak race condition na statusie joba.

### Negatywne
- **Brak skalowalności dla bardzo dużych raportów** — raport „wszyscy pacjenci, ostatnie 5 lat"
  może wyczerpać heap i wywalić cały pod K8s.
- Reverse-proxy (nginx/traefik) ma typowe `proxy_read_timeout 60s` — generacja >60s
  zwraca 504 do klienta, mimo że backend skończy generację.
- Brak audytu „kto pobrał jaki raport" — nie ma trwałego rekordu joba.

### Plan łagodzenia (krótkoterminowo, bez czekania na trigger)
- Hard-limit na zakres dat raportu: **max 12 miesięcy** w warstwie walidacji DTO.
- Hard-limit na liczbę wierszy: **max 50 000** — przy przekroczeniu zwracamy `413 Payload Too Large`
  z komunikatem „Skróć zakres dat".
- Logować `kptest_report_export_size_bytes` od dnia wdrożenia, żeby mieć dane do decyzji.
