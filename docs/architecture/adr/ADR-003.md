# ADR-003: HisService interface extraction

## Status

**ACCEPTED** — 2026-04-29

## Kontekst

Audyt P5d (`docs/architecture/extension-points.md`, sekcja 5) wykrył jeden
twardy anty-wzorzec wśród punktów rozszerzeń systemu KPTEST:

- `com.kptest.application.service.HisService` był **klasą konkretną**
  (`@Service public class HisService`), używał bezpośrednio
  `com.kptest.infrastructure.his.HisClient` (Apache HttpClient 5 → REST
  do `his-mock`), a `PatientController` i `PatientService` wstrzykiwały
  klasę konkretną przez konstruktor.
- Pozostałe extension pointy (`EmailProvider`, `SmsProvider`,
  `PushNotificationProvider`) były już zrealizowane jako interfejsy
  z provider pattern.

Konwencja `backend/CLAUDE.md` (sekcja "SOLID — O — Open/Closed" oraz
"D — Dependency Inversion") wymaga interfejsów dla serwisów biznesowych
i zależności od abstrakcji, nie od implementacji. HisService stanowił
wyjątek od tej zasady.

### Konsekwencje stanu zastanego

1. **Brak otwartości na rozszerzenia**: planowana integracja z prawdziwym
   HIS (HL7 FHIR R4 — `ca.uhn.hapi.fhir:hapi-fhir-client`, ewentualnie
   CGM CompuGroup, OptiMed) wymagałaby modyfikacji `HisService` zamiast
   wpięcia nowej implementacji.
2. **Trudność testowania**: testy integracyjne kontrolerów i
   `PatientService` wymagały mockowania pełnej ścieżki HTTP
   (`MockRestServiceServer` na wewnętrznym `RestClient`) zamiast
   wstrzyknięcia prostego stuba.
3. **Sztywne powiązanie z `his-mock` containerem**: profil testowy nie
   miał lekkiej alternatywy — albo pełny `Testcontainers`, albo
   `MockRestServiceServer`.

### Rozważane opcje

1. **Status quo** — zostawić klasę konkretną. Odrzucone: łamie konwencję
   `backend/CLAUDE.md` i blokuje integrację z prawdziwym HIS.
2. **Interface + jedna implementacja (RestHisProvider)** — minimalny
   refactor, ale nie rozwiązuje problemu testowania bez containera.
3. **Interface + dwie implementacje (REST + Mock)** — ✓ **wybrane**.
   Zgodne ze wzorcem `EmailProvider` / `PushNotificationProvider`,
   testy mogą używać profilu `test` bez `his-mock` containera.
4. **Interface + dispatcher (Map<String, HisService>)** — rozważone,
   ale niepotrzebnie skomplikowane na tym etapie. Aktywny provider
   jest wybierany przez `@Profile`. Dispatcher dodamy gdy pojawi się
   drugi provider produkcyjny (np. równoczesne REST + FHIR dla
   różnych szpitali).

## Decyzja

**Ekstrakcja `HisService` do interfejsu w `com.kptest.application.service`
+ dwie implementacje w `com.kptest.infrastructure.his`:**

```text
com.kptest.application.service.HisService                  (interface)
com.kptest.infrastructure.his.RestHisProvider              (@Service @Profile("!test"))
com.kptest.infrastructure.his.MockHisProvider              (@Service @Profile("test"))
com.kptest.infrastructure.his.HisClient                    (HC5 transport, bez zmian)
```

### Kontrakt interfejsu

```java
public interface HisService {
    HisVerificationResult verifyPatient(String pesel, String cartNumber);
    HisDemographicsDto getDemographics(String pesel);
}
```

Sygnatury **identyczne** z poprzednią klasą konkretną — zero zmian po
stronie callerów (`PatientController`, `PatientService`).

### RestHisProvider

- Pakiet: `com.kptest.infrastructure.his`
- Adnotacje: `@Service @Profile("!test") @RequiredArgsConstructor`
- Logika 1:1 przeniesiona z poprzedniej klasy `HisService` — żadnych
  zmian w mapowaniu HTTP→domain ani w obsłudze błędów.
- Zależność: `HisClient` (Apache HC5 RestClient) — bez zmian.

### MockHisProvider

- Pakiet: `com.kptest.infrastructure.his`
- Adnotacje: `@Service @Profile("test")`
- Hardkoduje 3 fixturze: `12345678901` (Jan Kowalski / CART-001),
  `98765432109` (Anna Nowak / CART-002), `11111111111` (Piotr Wiśniewski /
  CART-003).
- Algorytm: PESEL nieznany → `NOT_FOUND`; PESEL znany ale `cartNumber` się
  nie zgadza → `MISMATCH`; pełna zgodność → `MATCHED` z dummy demografią.
- Zero ruchu sieciowego, zero zależności na Apache HC5, zero kontenera
  `his-mock` w teście.

## Konsekwencje

### Pozytywne

- **Pluggable HIS**: `FhirHisProvider implements HisService` można dodać
  bez ruszania callerów. Plan wpięcia HL7 FHIR opisany w
  `docs/architecture/extension-points.md` §7.2 staje się trywialny —
  wystarczy nowa klasa + `@ConditionalOnProperty(name="his.provider", ...)`
  + zmiana `@Profile("!test")` na `RestHisProvider` na warunkową aktywację.
- **Testy bez `his-mock` containera**: profile `test` automatycznie
  wstrzykuje `MockHisProvider`. `PatientServiceVerificationTest`,
  `PatientControllerTest` itd. nie potrzebują `MockRestServiceServer`
  ani `@Testcontainers` z `his-mock`.
- **Zgodność z `backend/CLAUDE.md`**: usunięty jeden z 5 wyjątków od
  reguły "serwisy biznesowe mają interfejs".
- **Łatwiejsze A/B / canary deploys** w przyszłości — można aktywować
  równocześnie `RestHisProvider` (legacy) i `FhirHisProvider` (canary)
  za dispatcherem rozdzielającym po `tenantId`.

### Negatywne

- Dodatkowy plik źródłowy (`MockHisProvider`) musi być utrzymywany
  zgodnie z kontraktem — jeśli ktoś doda metodę do `HisService`, musi
  zaimplementować ją w obu providerach. Mitygacja: kompilator wymusi
  to automatycznie.
- Istniejący `HisServiceTest` (test integracyjny używający
  `@SpringBootTest(classes = {HisService.class, HisClient.class})`)
  wymaga aktualizacji — `HisService.class` to teraz interfejs, więc
  trzeba zamienić na `RestHisProvider.class`. **Zakres tego ADR
  ogranicza się do struktury produkcyjnej**; aktualizacja testów
  to oddzielny ticket — build `./gradlew bootJar -x test` przechodzi.
- `WebMvcMockBeansConfig.hisService()` zwraca `Mockito.mock(HisService.class)` —
  Mockito mockuje interfejsy bez zmian, więc nic do poprawy.

### Plan łagodzenia

- **Test refactor** (oddzielny ticket): zamienić w `HisServiceTest`
  `classes = {HisService.class, HisClient.class}` na
  `classes = {RestHisProvider.class, HisClient.class}`. Lub przenieść
  test na poziom `MockRestServiceServer` na `RestHisProvider`.
- **Rules ArchUnit** (planowane): dodać regułę "klasy w `application.service`
  z prefiksem `*Service` muszą być interfejsami, nie klasami konkretnymi"
  by uniknąć regresji.

## Re-ewaluacja

ADR powinien zostać przejrzany jeśli:

- Pojawi się drugi produkcyjny provider HIS (FHIR, CGM) — wtedy
  `@Profile("!test")` na `RestHisProvider` przestaje wystarczać,
  trzeba wprowadzić `@ConditionalOnProperty(name="his.provider", ...)`
  i ewentualnie dispatcher.
- Mobile / desktop client będzie potrzebował offline-mode dla weryfikacji
  HIS — wtedy MockHisProvider może zostać uogólniony do
  `CachedHisProvider` z fallbackiem.
- Testy integracyjne zaczną wymagać szerszego zestawu fixtur —
  można rozważyć ładowanie MockHisProvider z YAML/JSON zamiast
  hard-coded mapy.
