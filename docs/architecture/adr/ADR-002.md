# ADR-002: SecurityContext over @AuthenticationPrincipal

## Status

**ACCEPTED** — 2026-04-29

## Kontekst

W kodzie kontrolerów istnieją **dwa równoległe wzorce** odczytu zalogowanego użytkownika
(potwierdzone audytem `backend/src/main/java/com/kptest/api/controller/`):

### Wzorzec A — `SecurityContextHolder` (dominujący)

```java
// PatientController, MessageController, ProjectController, MaterialController, ...
import org.springframework.security.core.context.SecurityContextHolder;

@PostMapping
public ResponseEntity<MessageDto> sendMessage(@Valid @RequestBody SendMessageRequest req) {
    Authentication auth = SecurityContextHolder.getContext().getAuthentication();
    String userId = auth.getName();
    return ResponseEntity.ok(messageService.send(UUID.fromString(userId), req));
}
```

### Wzorzec B — `@AuthenticationPrincipal` (mniejszościowy)

```java
// AuthController, NotificationController
import org.springframework.security.core.annotation.AuthenticationPrincipal;

@PostMapping("/change-password")
public ResponseEntity<Void> changePassword(
        @AuthenticationPrincipal String userIdStr,
        @Valid @RequestBody ChangePasswordRequest req) {
    authenticationService.changePassword(UUID.fromString(userIdStr), req);
    return ResponseEntity.noContent().build();
}
```

Stan obecny (zliczone wystąpienia):
- `SecurityContextHolder.getContext().getAuthentication()` — używane w **8+** kontrolerach.
- `@AuthenticationPrincipal` — używane w `AuthController` i `NotificationController`.

JWT filter (`com.kptest.infrastructure.security.JwtAuthenticationFilter`) ustawia
principal jako `String userId` (UUID), nie pełny `UserDetails`. Dlatego oba wzorce
zwracają string i wymagają `UUID.fromString(...)`.

**Rozważane opcje**:

1. **Status quo** — zostawić mieszankę, dokumentować obie ścieżki.
2. **Standaryzacja na `SecurityContextHolder`** (większość kodu) — refactor 2 kontrolerów.
3. **Standaryzacja na `@AuthenticationPrincipal`** (idiom Springa, lepszy dla testów) —
   refactor 8+ kontrolerów + helper `UserPrincipalArgumentResolver`.
4. **Wprowadzenie `@CurrentUser` aspect** — własna anotacja inżektująca `User` (z DB
   lookupem) przez `HandlerMethodArgumentResolver`. Najpotężniejsze, ale dodaje magię.

## Decyzja

**Trzymamy się Wzorca A (`SecurityContextHolder.getContext().getAuthentication().getName()`)
jako oficjalnego standardu projektu.** Nowo dodawane kontrolery MUSZĄ używać tego wzorca.

Motywacja:
- **Spójność > teoretyczna czystość**. 80% kodu już używa wzorca A; refactor B → A
  to ~30 minut, B → A pełnym kierunkiem to ~3 dni z testami.
- Obecny `JwtAuthenticationFilter` nie zwraca pełnego `UserDetails`, tylko `String`,
  więc `@AuthenticationPrincipal` i tak nie daje dużej przewagi typowej.
- Helper utility (do dodania): `com.kptest.infrastructure.security.SecurityUtils#currentUserId()`
  pakujące boilerplate w jeden wywołanie.

Plan dorównania:
- Dodać `SecurityUtils.currentUserId(): UUID` w `infrastructure/security/`.
- W kolejnym sprintcie (nie w P5d — to dokument, nie refactor) zmienić
  `AuthController` i `NotificationController` z `@AuthenticationPrincipal` na
  `SecurityUtils.currentUserId()`.
- Dodać Checkstyle/ArchUnit rule blokującą import `@AuthenticationPrincipal` w nowych klasach.

```java
// Helper do dodania:
package com.kptest.infrastructure.security;

import org.springframework.security.core.Authentication;
import org.springframework.security.core.context.SecurityContextHolder;

public final class SecurityUtils {
    private SecurityUtils() {}

    public static UUID currentUserId() {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth == null || !auth.isAuthenticated()) {
            throw new UnauthorizedException("No authenticated user in context");
        }
        return UUID.fromString(auth.getName());
    }

    public static String currentUserIdString() {
        return SecurityContextHolder.getContext().getAuthentication().getName();
    }
}
```

## Konsekwencje

### Pozytywne
- Spójność stylistyczna — jedna konwencja, łatwiej code review.
- Łatwa migracja w testach integracyjnych — `@WithMockUser(username="<uuid>")` ustawia
  `authentication.getName()` automatycznie.
- Brak dependency na Spring `HandlerMethodArgumentResolver` (lżejsza konfiguracja).

### Negatywne
- **Mniej testowalne w testach jednostkowych** — controller jednostkowo testowany
  bez Spring Security context wymaga `SecurityContextHolder.setContext(...)` w `@BeforeEach`,
  co jest bardziej intruzywne niż wstrzyknięcie parametru przez `@AuthenticationPrincipal`.
- ThreadLocal `SecurityContextHolder` wymaga uważności w wątkach asynchronicznych
  (`@Async`, `CompletableFuture`) — trzeba używać `DelegatingSecurityContextRunnable`.
- Niezgodność z idiomami Spring Security 6 (gdzie zalecany jest `@AuthenticationPrincipal`
  z mapperem do własnego typu `User`).

### Plan łagodzenia
- W testach: stworzyć abstract base class `AbstractSecuredControllerTest` z `@BeforeEach`
  ustawiającym mock authentication.
- Dla `@Async` use case: utility `SecurityUtils.runAsCurrentUser(Runnable)`
  (`DelegatingSecurityContextRunnable`).
- Dokumentacja: dopisać sekcję „Authentication w kontrolerach" w `backend/CLAUDE.md`.

## Re-ewaluacja

ADR powinien zostać przejrzany jeśli:
- Wprowadzimy `@AuthenticationPrincipal User` z customowym mapperem (po dodaniu OAuth — zob. `extension-points.md` §7.1).
- Migrujemy do Spring Security 7 lub WebFlux (gdzie ThreadLocal nie działa, trzeba użyć `ReactiveSecurityContextHolder`).
- Dodamy reactive endpoints — wtedy obecny wzorzec przestaje działać i konieczna jest
  pełna migracja.
