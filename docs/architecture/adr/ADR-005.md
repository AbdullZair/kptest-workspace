# ADR-005: OpenAPI codegen — shared types portal ↔ mobile

## Status

**ACCEPTED** — 2026-04-29

## Kontekst

Audyt P5d (raport `RAPORT_GAP_ANALYSIS.md`, sekcja "Portal ↔ Mobile contract drift")
wykrył, że portal (`frontend/`) i mobile (`mobile/`) trzymają **niezależne, ręcznie
pisane** typy DTO dla każdego endpointu backendu (`Project`, `PatientDto`,
`MessageDto`, `TherapyStageDto`, ...). Wzorzec ten:

- powoduje **drift** — backend zmienia pole z `compliance_threshold: int` na
  `compliance_threshold: BigDecimal`, frontend nadal trzyma `number`, mobile trzyma
  `number`, ale prawdziwy kontrakt różni się i bug pojawia się w runtime;
- generuje **nieosynchronizowane typy** — `frontend/src/features/projects/types.ts`
  ma pole `Project.startDate`, `mobile/features/projects/types.ts` ma `start_date`,
  bo każdy zespół tłumaczy snake_case→camelCase inaczej;
- wymaga **podwójnego ręcznego utrzymania** każdej zmiany kontraktu (backend → portal
  + backend → mobile = 3 równoległe edycje na każdy nowy endpoint).

Backend wystawia już springdoc-openapi (`springdoc-openapi-starter-webmvc-ui:2.3.0`,
`/v3/api-docs` zwraca `openapi: 3.0.1`, **129 paths, 122 schemas** w aktualnym snapshot).

**Rozważane opcje**:

1. **Status quo** — utrzymywać typy ręcznie, polegać na PR review.
   Niska pewność, wysoki koszt długoterminowy.
2. **`openapi-typescript`** — czyste typy TypeScript, brak runtime, output ~8000 linii.
   Maintainerem jest tym samym co `openapi-fetch`, projekt aktywny.
3. **`@openapitools/openapi-generator-cli`** (Java-based, wymaga JRE) — generuje
   pełne klienty (axios/fetch + typy). Cięższy, ale daje od razu service layer.
4. **`orval`** — generuje hooki RTK Query / React Query bezpośrednio. Najwięcej
   automatyzacji, ale silne sprzężenie z generatorem.

## Decyzja

**Wybieramy opcję 2 — `openapi-typescript`**:

- frontend i mobile dostają skrypty `npm run generate:api` (live z localhost:8080)
  oraz `npm run generate:api:offline` (z `frontend/openapi.json` snapshot).
- Output zapisywany w `<app>/src/shared/api/generated/types.ts` (committed do repo —
  developer i CI mają deterministyczny snapshot kontraktu).
- Snapshot OpenAPI (`frontend/openapi.json`) committed jako pojedyncze źródło prawdy
  dla mobile (mobile generuje z `../frontend/openapi.json`).
- Adopcja **incremental** — istniejące typy lokalne (`features/projects/types.ts`)
  pozostają, nowe miejsca importują `components['schemas']['ProjectResponse']`
  zamiast pisać własne.

Motywacja:

- **Zero runtime impact** — `openapi-typescript` produkuje wyłącznie typy
  (kompilowane do `--noEmit`). Bundle size = 0.
- **Single source of truth** — backend springdoc → JSON spec → typy TS dla obu
  klientów. Backend zmienia DTO → `npm run generate:api` → typy automatycznie
  aktualizowane → `tsc --noEmit` wykryje wszystkie use site.
- **Brak sprzężenia z RTK Query / fetch** — niezależne od warstwy transportowej,
  więc frontend (RTK Query) i mobile (axios + RTK Query) mogą używać tych samych typów.
- Opcja 3/4 (orval, openapi-generator) były odrzucone bo wymuszają strukturę kodu
  (hooki, klienty), co kolidowałoby z istniejącą architekturą Feature-Sliced Design.

## Konsekwencje

### Pozytywne

- **Type safety end-to-end** — zmiana w backendzie DTO łamie kompilację frontend
  i mobile w jasnych miejscach.
- **Eliminacja driftu** — żadnych więcej rozbieżności typów `Project.startDate`
  vs `Project.start_date`.
- **Dokumentacja jako kod** — każda zmiana spec generuje diff w
  `src/shared/api/generated/types.ts` widoczny w PR.
- **Zero bundle cost** — typy są strip'owane w build.

### Negatywne

- **Manual refresh** — developer musi pamiętać `npm run generate:api` po zmianie
  backendu (mitigation: CI step + pre-commit hook — TODO follow-up).
- **Generated file w repo** — ~8155 linii w `types.ts` (mitigation: plik jest
  banner'owany "Do not make direct changes", `.eslintignore` może go wykluczyć).
- **snake_case vs camelCase** — backend zwraca `start_date`, generowane typy mają
  `start_date`. Istniejące lokalne typy używają `startDate` — refaktor adopcji
  wymaga decyzji per-feature (mitigation: incremental migration, oba istnieją
  równolegle).
- **OpenAPI snapshot vs live** — `frontend/openapi.json` może się rozjeść z
  backendem. Dlatego skrypt domyślny (`generate:api`) idzie live na localhost:8080,
  a `:offline` używa snapshot tylko gdy backend niedostępny.

### Plan łagodzenia

- **CI step** (TODO follow-up): GitHub Actions `verify-api-types`:
  - `cd backend && ./gradlew bootRun &` → wait for `/v3/api-docs`
  - `cd frontend && npm run generate:api && git diff --exit-code`
  - jeśli diff → fail build z komunikatem "Run npm run generate:api".
- **Pre-commit hook** (opcjonalnie) w `.husky/pre-commit` — wywołuje generate:api
  jeśli zmienił się `backend/src/main/java/com/kptest/api/`.
- **Adopcja incremental**: nowe RTK Query slices używają generowanych typów,
  istniejące migrują przy najbliższym refaktoringu danego feature.

## Re-ewaluacja

ADR powinien zostać przejrzany jeśli:

- Backend przejdzie z springdoc na ręczny `OpenApiCustomizer` z odbiegającymi schematami.
- Pojawi się wymóg generacji klientów dla 3+ języków (Python/Go/Kotlin) — wtedy
  warto rozważyć przejście na `openapi-generator-cli`.
- Bundler zacznie zgłaszać problem z dużym plikiem `types.ts` (>500KB) — split per-tag.
- Powstanie GraphQL — wtedy codegen przejdzie na schema-first GraphQL Codegen.

## Implementacja (B3)

### Frontend

```json
// frontend/package.json scripts
"generate:api": "openapi-typescript http://localhost:8080/v3/api-docs --output src/shared/api/generated/types.ts",
"generate:api:offline": "openapi-typescript ./openapi.json --output src/shared/api/generated/types.ts"
```

Sample użycia (incremental adoption pattern):

```typescript
// frontend/src/features/projects/api/projectApi.ts
import type { components } from '@shared/api/generated/types'

export type GeneratedProjectResponse = components['schemas']['ProjectResponse']
// type for endpoint response payload (alternative pattern):
// type R = operations['getProjectById']['responses']['200']['content']['application/json']
```

### Mobile

```json
// mobile/package.json scripts
"generate:api": "openapi-typescript http://localhost:8080/v3/api-docs --output src/shared/api/generated/types.ts",
"generate:api:offline": "openapi-typescript ../frontend/openapi.json --output src/shared/api/generated/types.ts"
```

Mobile reuse'uje `frontend/openapi.json` (committed snapshot) żeby uniknąć
duplikacji backupów — single source of truth.

### Verification

- `frontend/src/shared/api/generated/types.ts` — 8155 linii, 122 schemas, 129 paths.
- `mobile/src/shared/api/generated/types.ts` — identyczna treść (ten sam spec).
- `cd frontend && npx tsc --noEmit` → 0 nowych błędów (tylko 2 pre-existing
  błędy w `RegisterPage.tsx` niezwiązane z B3).
